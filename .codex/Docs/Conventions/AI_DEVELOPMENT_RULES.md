# AI 개발 규칙

## 작업 우선순위

1. 현재 대화의 직접 프롬프트
2. `.codex/WORK_ORDER.md`
3. `.codex/AGENTS.md`
4. `.codex/Docs/Conventions/*`
5. `.codex/Docs/Planning/*`
6. 기존 프로젝트 구조

## 기본 원칙

- 확인되지 않은 정보는 만들지 않는다.
- 현재 프로젝트 파일에서 확인한 내용만 기준으로 삼는다.
- 확정된 기획 범위는 `PROJECT_PLAN.md`를 따른다.
- 미확정 항목은 씬 구현 방식, 저장 시스템, 전체 UI 구조, 세부 아트 스타일을 강제하지 않는다.
- 초반부터 완성형 구조를 설계하려 하지 말고, 구현하면서 확인된 내용만 문서에 반영한다.
- 기존 구조를 우선 존중한다.
- 변경 범위는 요청 해결에 필요한 최소 범위로 제한한다.
- 사용자가 수정한 파일은 임의로 복구하거나 되돌리지 않는다.
- 명시 요청 없는 대규모 리팩터링, 폴더 재배치, 이름 변경은 하지 않는다.

## 구현 원칙

- Unity 기본 생명주기와 직관적인 MonoBehaviour 구조를 우선한다.
- 불필요한 매니저, 싱글톤, 추상화를 남발하지 않는다.
- 반복되는 로직만 공통화한다.
- ScriptableObject는 실제 재사용 가치가 있을 때만 추가한다.
- public API 변경은 참조 범위와 영향도를 확인한 뒤 최소화한다.
- 매직 넘버는 의미 있는 상수, SerializedField, 설정 데이터로 분리한다.
- 함수는 한 가지 역할을 우선한다.
- 새 시스템을 만들기 전에 기존 스크립트와 씬 사용 방식을 확인한다.
- 아직 존재하지 않는 씬, 시스템, 오브젝트를 전제로 코드를 작성하지 않는다.

## Unity 작업 주의사항

- `.meta` 파일은 Unity GUID 참조를 유지하는 핵심 파일로 취급한다.
- 씬, 프리팹, ScriptableObject, Animator, Timeline 참조가 깨질 수 있는 변경을 조심한다.
- 파일/폴더 이동 또는 이름 변경 전 다음을 확인한다.
  - [ ] 참조하는 씬/프리팹이 있는가
  - [ ] 인스펙터에 연결된 SerializedField가 있는가
  - [ ] Addressables, Resources, 빌드 세팅, 스크립트 참조가 있는가
- 인스펙터 연결값을 코드에서 요구할 경우 null 처리 또는 명확한 오류 메시지를 둔다.
- Unity 에디터에서만 동작하는 코드는 `Editor` 폴더 또는 조건부 컴파일을 사용한다.
- 패키지 추가는 필요성이 명확할 때만 한다.

## 검증 규칙

- 가능한 경우 변경 후 아래 중 해당 항목을 수행한다.
  - [ ] Unity 컴파일 확인
  - [ ] Play Mode 수동 확인
  - [ ] 관련 씬 열기 확인
  - [ ] EditMode/PlayMode 테스트
  - [ ] 코드 포맷/정적 오류 확인
- 테스트하지 못했으면 결과 보고에 이유를 남긴다.
- 검증 실패를 숨기지 않는다.

## 작업 기록 규칙

- 작업 결과에는 다음을 짧게 기록한다.
  - 변경 파일
  - 핵심 변경 내용
  - 검증 내용
  - 미검증 항목과 이유
- 확정된 의사결정만 `.codex/Docs/DECISIONS.md`에 기록한다.
- 미확정 기획이나 추측은 결정 기록에 넣지 않는다.

## 금지사항

- 확인되지 않은 기획을 문서나 코드에 반영
- 사용자 변경 파일 임의 복구
- 요청 없는 public API 변경
- 요청 없는 씬/프리팹 구조 변경
- 요청 없는 대규모 폴더 재구성
- 테스트하지 않은 내용을 검증 완료처럼 작성
