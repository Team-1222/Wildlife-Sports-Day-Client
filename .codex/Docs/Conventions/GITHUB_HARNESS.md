# GitHub 하네스

## 운영 기준

- 현재 기준: 1인 클라이언트 개발 흐름
- 과한 Git Flow를 강제하지 않는다.
- 기본은 `main`, `develop` 중심으로 단순하게 운영한다.
- 작은 작업은 브랜치 없이 `develop`에서 직접 작업 가능하다.
- 범위가 크거나 되돌릴 가능성이 있으면 작업 브랜치를 만든다.

## 브랜치 전략

- `main`: 안정 버전. 실행/배포 가능한 상태 유지.
- `develop`: 일상 개발 통합. 작은 기능, 문서, 일반 수정은 여기서 처리 가능.
- `feature/*`: 기능 개발. 작업 규모가 크거나 분리가 필요할 때만 사용.
- `fix/*`: 버그 수정. 원인 추적/검증이 필요한 수정에 사용.
- `hotfix/*`: 긴급 수정. `main` 기준 긴급 대응에 사용.
- `docs/*`: 문서 작업. 문서만 변경하고 분리가 필요할 때 사용.
- `refactor/*`: 구조 개선. 영향 범위가 넓을 때 사용.
- `test/*`: 테스트 추가/검증 전용 작업에 사용.

예시:

- `feature/player-movement`
- `fix/ui-overflow`
- `docs/codex-harness`

## 커밋 메시지 규칙

- 커밋 메시지는 한글로 작성한다.
- 기본 형식: `타입: 한글 설명`
- 이슈가 있으면 끝에 `#이슈번호`를 붙인다.
- 한 커밋에는 가능한 한 한 가지 목적만 담는다.

예시:

- `feat: 플레이어 이동 입력 추가`
- `fix: 버튼 입력 중복 실행 수정 #12`
- `docs: AI 작업 하네스 문서 보강`
- `refactor: 초기화 흐름 정리`

## 커밋 타입

- `feat`: 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 수정
- `test`: 테스트 추가/수정
- `refactor`: 동작 변경 없는 구조 개선
- `build`: 빌드/패키지/설정 변경
- `style`: 포맷, 공백 등 동작 영향 없는 수정
- `rename`: 파일/폴더 이름 변경
- `remove`: 파일 삭제
- `chore`: 기타 관리 작업

## 이슈 규칙

- 필요한 작업만 이슈로 만든다.
- 혼자 처리할 작은 작업은 TODO만으로 관리해도 된다.
- 제목 형식: `[Type] 작업 설명`

예시:

- `[Feature] 플레이어 이동 추가`
- `[Fix] UI 텍스트 넘침 수정`
- `[Docs] 프로젝트 기획서 템플릿 정리`

## PR 규칙

- 1인 개발에서는 PR을 필수로 강제하지 않는다.
- `main` 반영 전 검토가 필요하거나 작업 규모가 크면 PR을 사용한다.
- PR 제목 형식: `[Type] 작업 설명`
- PR 본문에는 검증 결과와 미검증 항목을 반드시 적는다.

PR 제목 예시:

- `[Feature] 플레이어 이동 추가`
- `[Fix] 버튼 입력 중복 실행 수정`
- `[Docs] 하네스 문서 보강`

## PR 템플릿 예시

```md
## 개요

- 작업 목적:
- 작업 범위:

## 변경한 내용

- 

## 기타 사항

- 미검증 항목:
- 확인 필요:
```

## main 반영 전 체크리스트

- [ ] Unity에서 컴파일 오류가 없는가
- [ ] 주요 씬이 열리는가
- [ ] 변경한 기능을 직접 확인했는가
- [ ] 불필요한 `.meta` 변경이 없는가
- [ ] 임시 파일/로그가 포함되지 않았는가
